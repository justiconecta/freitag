## Contexto

Usuarios do NormaChat reportam que a IA nao encontra informacoes que **existem** no banco vetorial. Exemplo concreto: ao perguntar "Qual analise para determinar DQO?" ou "O que pode me explicar sobre a secao 5220, do Standard Methods?", o sistema retorna paginas erradas (127, 128, 475) e diz nao ter encontrado a informacao, embora a secao 5220 do Standard Methods esteja indexada no banco.

Investigacao tecnica do PM identificou 7 problemas na pipeline RAG, variando de critico a baixo. Esta story cobre todas as correcoes necessarias para restaurar a confiabilidade da busca.

**Pipeline ativo em producao:** Frontend Next.js (`frontend/src/app/api/chat/route.ts` -> `frontend/src/lib/services/rag.ts`), NAO o backend Python.

---

## Problema Raiz

A busca vetorial atual sofre de multiplos problemas que degradam o recall (capacidade de encontrar documentos relevantes):

1. Indice IVFFlat com `probes=1` (default) pesquisa apenas ~1% dos dados
2. Chunking por pagina sem continuidade entre paginas quebra secoes
3. SDKs diferentes entre ingestao (Python) e busca (JS) podem gerar vetores inconsistentes
4. Sem `task_type` nos embeddings (Gemini suporta RETRIEVAL_DOCUMENT/RETRIEVAL_QUERY)
5. Queries curtas/siglas vao direto para embedding sem expansao
6. Busca puramente vetorial, sem full-text search como complemento
7. IVFFlat tem recall inferior ao HNSW para este caso de uso

---

## User Story

**Como** analista de laboratorio da Freitag,
**Eu quero** que o NormaChat encontre informacoes relevantes nas normas tecnicas de forma confiavel,
**Para que** eu possa consultar qualquer secao, metodo ou parametro sem que o sistema diga falsamente que nao encontrou a informacao.

## Technical Notes

### Ordem de Implementacao Recomendada
1. **AC1** (HNSW) + **AC7** (threshold) — correcoes rapidas no banco, impacto imediato
2. **AC2** (busca hibrida) — nova funcao SQL + updates nos servicos
3. **AC3** (chunking) + **AC4** (task_type/SDK) — correcoes na ingestao
4. **AC6** (re-ingestao) — deve rodar APOS AC3+AC4
5. **AC5** (query expansion) — melhoria adicional, pode ser feita em paralelo
6. **AC8** (validacao) — final, depois de tudo aplicado

### Migration SQL (AC1 + AC2 + AC7)
Deve ser criada uma unica migration Supabase em `supabase/migrations/` que:
1. Remove indice IVFFlat
2. Cria indice HNSW
3. Adiciona coluna `fts` com trigger
4. Cria funcao `hybrid_search_chunks`
5. Atualiza default do `search_chunks`

### Impacto na Re-ingestao
AC3 (chunking cross-page) e AC4 (task_type) alteram como os embeddings sao gerados. Isso significa que **todos os documentos precisam ser re-ingestados** apos essas mudancas. A re-ingestao leva tempo devido ao rate limiting do Google free tier (~1500 RPD).

### Dual Pipeline (Frontend + Backend)
O pipeline ativo e o frontend Next.js, mas o backend Python deve ser mantido sincronizado para consistencia. Todas as correcoes de servico devem ser aplicadas em ambos os pipelines.

## Risks

| Risco | Mitigacao |
|-------|----------|
| Re-ingestao demora devido a rate limiting | Usar checkpoint/resume existente no script |
| HNSW usa mais memoria que IVFFlat | Volume de dados atual e pequeno (~6 PDFs), impacto negligivel |
| Query expansion adiciona latencia | Usar modelo rapido (Haiku), timeout de 3s |
| Busca hibrida pode trazer mais ruido | Score ponderado (0.7 vetor + 0.3 FTS) prioriza relevancia semantica |
